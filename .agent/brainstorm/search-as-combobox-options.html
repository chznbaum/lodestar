Lodestar · Vesper · search study

Make search feel like the combobox

The five filter pills (Status / Domain / Remote / Size / Stage) are the polished Combobox: type-to-filter with tiered ranking (prefix › substring › alias), a gold highlight on the matched run, and keyboard nav. The free-text company search next to them is, today, a plain <input> driving a flat .includes() match over name + domain + notes — no ranking, no highlight, no keyboard. This study shows three directions for closing that gap, each across its states (resting · typing · no-match), so we can pick one. Demo query throughout: sec.

B

Typeahead popover

navigate · new surface

The most literal port of the combobox: typing opens a dropdown of matching companies under the search box — monogram, name, a sub-line, ranked, gold-highlighted, with the matched reason. ↑↓ moves the active row, ↵ opens that company. It reads as a command-palette / quick-jump. The tension: this is navigate, not filter, and it adds a second result surface on top of the inline list — we'd have to decide how the two relate (does the list also filter? does the popover replace it?).

The decision

The directions nest: A ⊂ C (C is A plus the keyboard model), while B is the orthogonal “new popover surface” choice. So the call is really two questions: (1) inline or popover? and, if inline, (2) with keyboard nav or without?

My recommendation: C. It delivers the two things that make the combobox's type-to-filter feel good — relevance-ranked, highlighted matches and a keyboard-first rhythm — while staying a single surface that reuses the list we already have. A is a clean fallback if keyboard nav feels like scope creep (and C degrades to A gracefully). B's quick-jump is genuinely nice, but it's a command-palette — a different, navigate-first feature — and I'd rather propose it later as its own thing (maybe a global ⌘K) than bolt a second result surface onto this page now.

One open question either way (flagged on C): while a query is active, should the list collapse to a single flat, relevance-ranked result set — ignoring the Queue/By-domain grouping and the chosen sort? Ranking is meaningless inside a domain group, so I lean yes: searching enters a temporary “results mode,” and clearing the query restores the tab's normal grouping/sort. Worth deciding before we plan.
