Roles tab — three directions

A redesign of the Roles tab inside the company workspace (companies/[slug]). Built on the live Vesper tokens (squared corners, aubergine primary, gold accent) and real class atoms. All three render the same fetched-role set for one startup (Lattice) so the IA differences are the only variable. Lifecycle: new → fetching (live) → scored → failed. Live phases use the real job_detail stages: jd-scrape → sanitize → structure-jd → gap-detect → research-gaps → fit-score → alignment. Bands: 80+ strong · 60–79 good · 40–59 partial · 20–39 weak · 0–19 mismatch/dealbroken.

Direction 1

Triage List

flat · sortable · row-is-state
Rationale. One dense, sortable list; each row carries its whole lifecycle (fit number, state chip, live step-strip, durable outcome). A sticky action bar holds selection + "Fetch selected" so the verb is always reachable while scrolling. Sorting by fit puts best prospects on top the instant scores land. Tradeoff: a long flat list buries band structure — you read scores, not groups; with many roles the eye has to scan to see "which are strong".

Direction 2

Fit-Banded Board

grouped-by-band · best-first · cards
Rationale. Roles are grouped into collapsible band sections (Strong → Dealbroken, plus a Needs-fetch group for unscored/new). Triage becomes "work the Strong band, skim Good, ignore the rest" — the structure does the prioritizing. Cards carry a fit badge + a pick checkbox; the band header offers "fetch all in this band". Dealbroken sits last, visually dampened. Tradeoff: band groups only become meaningful after scoring — before a fetch everything piles into one "Needs fetch" group, so the board looks empty/lopsided on a fresh discovery; also costs more vertical space per role than a list.

Direction 3

Split List + Run Console

master list · durable run panel
Rationale. A compact role list on the left; a persistent Run console on the right that durably records every fetch trigger — started (with run id + live step phase), skipped (already running), and failed-to-start (with reason) — so outcomes never vanish like a toast. The console is the single place to watch live progress and read what happened, decoupled from where rows scroll. Mirrors the existing Checks/run mental model. Tradeoff: splits attention across two regions and spends ~16rem of width on the console; for a quick "select 3, fetch, glance" it's heavier than inline state, and on narrow windows the console drops below the list.
